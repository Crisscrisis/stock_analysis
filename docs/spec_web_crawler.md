# 指数成分股数据采集器 — 产品需求规格

## 背景

当前系统的数据获取是「被动按需」模式：用户查看某只股票时才去拉取数据。这导致：
- 首次查看任意股票都有明显等待
- 没有全量数据支撑跨股票的分析和对比
- 无法在离线状态下浏览历史数据

需要一个主动采集机制，针对重要指数的成分股，每天自动收集并持久化全量数据。

## 覆盖范围

### 指数池

| 指数 | 市场 | 约覆盖股票数 |
|------|------|-------------|
| 纳斯达克 100 | 美股 | ~100 只 |
| 恒生指数 | 港股 | ~80 只 |
| 恒生科技指数 | 港股 | ~30 只 |

未来可能扩展至沪深 300、标普 500 等其他指数，设计上需便于添加。

### 采集数据类型

每只成分股采集以下信息：

| 数据类型 | 说明 |
|---------|------|
| OHLCV K线 | 每日开高低收量 |
| 基本面 | PE(TTM)、PB、总市值、营收(TTM)、净利润(TTM)、股息率 |
| 资金流向 | 北向资金（仅A股）、主力净流入 |
| 财报摘要 | 季度/年度营收、净利润、EPS 等关键财务指标 |
| 分红记录 | 历史分红/派息时间及金额 |

> 采集的数据类型取决于数据源可提供的内容，尽可能多地采集。

## 用户故事

### US-1: 按指数采集成分股数据

**作为** 一个投资者，
**我想要** 系统每天自动采集纳斯达克100、恒生指数、恒生科技指数中所有成分股的数据，
**以便** 我可以随时查看任意一只成分股的完整信息，而无需等待实时拉取。

**验收标准：**
- 三个指数的成分股列表能正确获取，且与最新官方公布的成分一致
- 每只成分股的 OHLCV、基本面、资金流向、财报、分红数据均被采集并存入数据库
- 每个指数的数据存储在各自独立的表中，互不混淆
- 采集完成后可通过现有 CLI 工具或 API 查询到新数据

### US-2: 按市场收盘时间分别定时采集

**作为** 一个投资者，
**我想要** 各市场在收盘后分别自动触发采集，
**以便** 我在每个市场收盘后尽快获得当天最新数据。

**验收标准：**
- 港股相关指数（恒生指数、恒生科技指数）在港股收盘后自动采集
- 美股相关指数（纳斯达克100）在美股收盘后自动采集
- 非交易日（周末、节假日）不执行采集
- 定时采集无需人工干预，程序运行期间自动触发

### US-3: 手动触发采集

**作为** 一个投资者，
**我想要** 能随时手动触发一次完整采集（或指定某个指数的采集），
**以便** 在定时采集之外，我可以按需刷新数据。

**验收标准：**
- 提供手动触发全部指数采集的方式
- 提供手动触发单个指数采集的方式
- 手动触发时显示采集进度
- 手动采集与定时采集使用相同的逻辑，结果一致

### US-4: 首次运行补采近期历史数据

**作为** 一个新用户，
**我想要** 首次运行采集器时自动补采近1年的历史数据，
**以便** 我在初始化后立即拥有足够的历史数据进行分析。

**验收标准：**
- 首次运行时检测到数据库为空（或某指数无数据），自动回溯采集近1年数据
- 后续每日运行仅采集增量数据（当天新数据）
- 补采过程中显示进度，允许中断后续运行时从断点继续

### US-5: 采集报告

**作为** 一个投资者，
**我想要** 每次采集完成后看到一份汇总报告，
**以便** 我了解采集是否成功、覆盖了多少只股票、耗时多久。

**验收标准：**
- 每次采集结束后输出汇总报告，包含：
  - 各指数采集的股票数量（成功/失败/跳过）
  - 总耗时
  - 失败的股票列表及失败原因
- 报告在终端输出，格式清晰易读

### US-6: 采集失败时静默跳过

**作为** 一个投资者，
**我想要** 某只股票采集失败时不影响其他股票的采集，
**以便** 单个股票的数据源异常不会导致整个采集任务中断。

**验收标准：**
- 单只股票采集失败时，记录错误日志并跳过，继续采集下一只
- 失败的股票出现在采集报告的失败列表中
- 不因单只股票的失败而中断整个采集流程
- 不对失败的股票进行自动重试

### US-7: 成分股调仓处理

**作为** 一个投资者，
**我想要** 当某只股票被移出指数时，系统保留其历史数据并标记为「非活跃」，
**以便** 我仍可查看历史数据，同时清楚知道它已不再是当前成分股。

**验收标准：**
- 每次采集时自动获取最新成分股列表
- 对比上次列表，识别新增和移出的股票
- 新增股票开始采集，并补采近期历史数据
- 移出的股票标记为非活跃状态，停止后续采集
- 非活跃股票的历史数据永久保留，仍可查询

## 边界情况

### 数据源相关
- **数据源临时不可用**：某个数据源（akshare/yfinance）暂时无法访问时，跳过受影响的股票，记录日志，不阻塞其他股票或其他数据源的采集
- **数据源返回空数据**：某只股票查询结果为空（如停牌、退市），记录日志并跳过，不覆盖已有数据
- **数据源限流/封禁**：请求频率过高被限流时，合理控制采集速率，避免触发封禁
- **数据源格式变更**：akshare/yfinance 接口返回的数据结构发生变化时，采集失败应有明确的错误信息

### 成分股列表相关
- **成分股列表获取失败**：无法获取最新成分股列表时，使用上次成功获取的列表继续采集，不能因为列表获取失败就跳过整个指数
- **港股同时属于恒生指数和恒生科技指数**：同一只股票出现在多个指数中，各指数表中独立存储，采集时避免对同一股票重复请求数据源
- **成分股数量异常**：若获取到的成分股数量与预期差异过大（如纳斯达克100只返回10只），应视为异常，记录警告并使用上次列表

### 时间与调度相关
- **跨时区处理**：三个市场处于不同时区，采集时间点需基于各市场当地时间计算
- **非交易日**：周末和各市场的法定节假日不执行采集
- **程序重启后的定时恢复**：程序重启后应自动恢复定时任务，如果当天尚未采集则补采
- **同一天重复运行**：如果当天已采集过，重复运行时应跳过或更新（幂等），不产生重复数据
- **补采期间遇到非交易日**：历史回溯时数据源可能自然跳过非交易日，不应视为异常

### 数据一致性相关
- **采集中途中断**：程序崩溃或被终止时，已成功写入的数据应保留，下次运行时可继续未完成的部分
- **并发采集冲突**：手动触发和定时触发同时发生时，应避免对同一只股票的重复采集
- **数据去重**：同一股票同一日期的数据不应出现重复记录
